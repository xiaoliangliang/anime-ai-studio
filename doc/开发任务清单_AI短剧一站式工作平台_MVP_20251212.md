## 0. MVP 范围与硬约束（必须遵守）
1. 仅 Web 桌面端；单人使用。
2. 数据存储：IndexedDB（项目元数据 + 媒体资产本地缓存）。
3. 图片必须：imgbb 云托管（供 AI API 引用外链）+ IndexedDB 本地缓存（可清理）。
4. 视频资产：仅 IndexedDB 本地存储（不可自动清理，需提示风险）。
5. 下载能力：
   - ✅ MVP：单张图片下载、单个镜头视频下载（从资源预览/详情入口触发）。
   - ⛔ MVP 不做：项目导出、批量下载（ZIP）、合成成片导出。
6. AI 输出：全链路 **JSON + Schema 校验**（Ajv + 重试），不依赖 Markdown/正则解析。
7. 分镜字段：景别/机位/运镜使用中文枚举；`duration` 统一为秒（number）。
8. 过期机制（Stale）：细粒度（集/场景/镜头级别），只标记受影响下游；孤儿数据不级联删除。
9. 导演阶段：MVP 仅“逐镜头生成 + 列表预览 + 单镜头下载”，时间线/音频/字幕/成片导出均为 MVP后。
10. **批量生成策略：MVP 仅支持串行逐条生成**（API 有并发限制），不做并行队列；并发控制延后到 MVP 后。

## 1. 任务编号与使用说明
- 任务 ID 形如：`T0.1`、`T7.2`。
- 每个任务包含：目标 / 产出 / 验收 / 依赖 / 并行建议。
- 并行开发建议：在完成 `T0.*`（基础脚手架与约束统一）后，后续可按“并行包”拆给多个编程 agent。

---

## 2. 依赖关系总览（文字版）
- `T0.*`（基础脚手架/依赖/约束统一）是所有任务的前置。
- 后端代理能力：`T1.*` 可与前端基础能力并行，但 **Chat 相关任务依赖 /api/chat**。
- 公共能力：
  - `T2.*`（AI JSON+Schema 校验链路）是阶段 1/2/3 的前置。
  - `T3.*`（数据模型 + stale）是所有阶段的数据写入/过期标记前置。
  - `T4.*`（IndexedDB 存储）是项目管理与资产落库前置。
  - `T6.*`（tldraw 画布与 shapes）是“画布渲染产出”前置。
- 业务阶段依赖链（数据层面）：
  - 编剧（T8.*）→ 分镜（T9.*）→ 图像设计（T10.*）→ 美工（T11.*）→ 导演（T12.*）
  - 实现层面可并行开发（用 mock 数据），但联调验收必须按上述顺序打通。

## 3. 并行开发拆包建议
完成 `T0.*` 后建议并行拆为 4 个包：

- 并行包 A（后端/代理）：`T1.1 ~ T1.4`
- 并行包 B（数据与存储）：`T3.*` + `T4.*`
- 并行包 C（AI JSON+Schema + prompts + ChatService）：`T2.*`
- 并行包 D（前端框架 + 画布基础）：`T5.*` + `T6.*`

当 A/B/C/D 基本完成后，再并行拆阶段：
- 并行包 E（阶段1 编剧）：`T8.*`
- 并行包 F（阶段2 分镜）：`T9.*`
- 并行包 G（阶段3 图像设计师）：`T10.*`
- 并行包 H（阶段4 美工）：`T11.*`
- 并行包 I（阶段5 导演）：`T12.*`

## 4. P0（MVP 必做）任务清单

### T0. 基础脚手架与约束统一（所有任务前置）
#### T0.1 依赖补齐与项目结构落地
目标：补齐 MVP 需要的依赖与目录骨架，使后续任务能按技术方案落文件。
产出（示例）：
- 增加依赖：`react-router-dom`、`idb`、`ajv`（以及必要类型）。
- 建立目录：`src/pages`、`src/components`、`src/contexts`、`src/hooks`、`src/services`、`src/types`、`src/prompts`、`src/utils`。
验收：项目能正常启动、构建，目录结构清晰。
依赖：无。
并行建议：必须先做。

#### T0.2 统一“密钥不进前端”的工程约束
目标：前端不再持有 Pollinations / imgbb key；统一通过 `server.js` 代理访问。
产出：
- 前端通过 `VITE_API_BASE_URL` 指定代理地址（默认 `http://localhost:3001`）。
- 清理前端直连与 query key 模式（按技术方案口径）。
验收：前端代码中不出现 `POLLINATIONS_API_KEY / IMGBB_API_KEY / VITE_API_KEY / VITE_IMGBB_KEY` 等。
依赖：T0.1。
并行建议：与 T1.* 可并行推进，但最终以“前端无密钥”为准。

### T1. Express 代理服务器（server.js）
#### T1.1 环境变量与密钥管理
目标：将 `server.js` 中硬编码密钥迁移为环境变量读取（允许保留 demo fallback，但需注释风险）。
产出：
- `POLLINATIONS_API_KEY`、`IMGBB_API_KEY`、`PORT` 等读取逻辑。
- README/注释说明如何配置。
验收：不配置 env 时也能按 demo 方式跑通（可选），配置 env 后优先使用 env。
依赖：T0.1。
并行建议：可与前端并行。

#### T1.2 新增 `/api/chat`（文生文）代理
目标：提供编剧/分镜/图像设计阶段的 Chat Completions 代理。
产出：
- `POST /api/chat` → `https://text.pollinations.ai/v1/chat/completions`
- 支持 `stream` 与非流式（最少非流式必做）。
验收：curl/postman 可调用成功；前端可通过该端点拿到 `choices[0].message.content`。
依赖：T1.1。
并行建议：与 T2.2/T2.3 并行。

#### T1.3 统一图片/视频代理端点行为
目标：确保 `/api/txt2img`、`/api/img2img`、`/api/img2video` 与技术方案口径一致：
- 使用 `gen.pollinations.ai`
- 正确转发 `Content-Type`（image/* 或 video/*）
产出：
- 错误处理（非 image/video 内容时返回可读错误）。
验收：三类端点均可稳定返回；img2video 返回 `video/*`。
依赖：T1.1。
并行建议：可与前端并行。

#### T1.4 `/api/health` 健康检查
目标：本地联调时可快速判断代理是否正常。
验收：访问返回 ok。
依赖：无。

### T2. AI JSON+Schema 校验链路（通用能力）
#### T2.1 `src/prompts/*.json` 体系落地
目标：将各阶段系统提示词文件纳入项目版本化管理，并在提示词中明确输出必须为 JSON 且符合 Schema。
产出：
- `src/prompts/screenwriter.json`
- `src/prompts/storyboard.json`
- `src/prompts/imageDesigner.json`
验收：提示词可被 ChatService 加载；包含 `output_format` 指向对应 Schema。

并行建议：可与 T1.* 并行。

#### T2.2 ChatService（前端）
目标：实现 `src/services/chat.ts`，统一走 `/api/chat`。
产出：
- `sendChatMessage()`（非流式必做）
- （可选）`streamChatMessage()`
验收：可在浏览器中成功拿到 AI 返回文本。
依赖：T1.2。

#### T2.3 Schema + Ajv 校验服务
目标：实现 `src/services/validation.ts`，注册并校验各阶段 Schema。
产出：
- ScreenwriterOutput / StoryboardOutput / ImageDesignerOutput 的 JSON Schema（可放 `src/schemas/*.json` 或 `src/types/aiSchemas.ts`）。
- `validateAIOutput(stage, outputType, data)`
验收：对正确 JSON 返回 valid=true；错误 JSON 返回 errors。
依赖：T0.1。
并行建议：可与 T2.2 并行。

#### T2.4 `callAIWithValidation`（重试 + 纠错提示）
目标：实现 `src/services/aiOutputHandler.ts`：
- parse JSON（兼容 ```json 代码块）
- Ajv 校验
- 自动重试（最多 3 次）并把错误反馈给 AI 要求修正
产出：`callAIWithValidation<T>()`
验收：对“输出格式不合规”的模型返回，能自动纠错重试并最终拿到合规 JSON（或返回可解释错误）。
依赖：T2.2 + T2.3。

### T3. 数据模型 + 过期机制（Stale）
#### T3.1 TypeScript 数据模型（Project/Stages/Asset）
目标：按技术方案落地核心类型：Project、各阶段数据、Asset 引用模型。
关键约束：
- 分镜：中文枚举（景别/机位/运镜），`duration` 为 number 秒。
- 图片/视频：业务数据引用 `assetId`，不在 JSON 内放 Data URL。
产出：`src/types/project.ts`、`src/types/stage.ts`、`src/types/assets.ts`（或等价拆分）。
验收：类型可被各阶段复用；无明显字段冲突。
依赖：T0.1。

#### T3.2 StaleTracker（按集/场景/镜头粒度）
目标：实现变更传播与 UI 过期标记能力。
产出：`src/services/staleTracker.ts`（或 utils）：
- `markDownstreamStale(project, entityType, entityId)`
- `checkStale(entity, sourceEntity)`
验收：上游数据更新后，下游关联数据被标记 `isStale=true`，且不误伤无关数据。
依赖：T3.1。
